02
Data Pages — dbo.Employees (50 rows → 5 Pages)
ஒரு data page-ல என்ன என்ன இருக்கும் — Header, Rows, Free Space, Offset Array — full skeleton
🏗️ DATA PAGE ANATOMY — ஒரு Page-ல என்ன structure இருக்கும்

① Header (96 bytes)
Page address (FileID:PageNum), page type, free space, slot count, prev/next page pointers — SQL Server internal metadata.
② Row Data
Actual table rows stored here. Each row has: null bitmap + column lengths + column values. Rows are NOT sorted in heap (they insert in any position with free space).
③ Free Space
Remaining space in the page. New rows go here. When full, SQL Server allocates a new page.
④ Slot Array (Offset Table)
Grows backwards from end of page. Each slot = 2 bytes = byte offset where a row starts. RID = FileID:PageNum:SlotOffset. This is how index pages point to exact rows.
📄 5 DATA PAGES — Employees table (10 rows per page, sorted by EmpID because Clustered Index exists)
💡 Clustered Index இருக்கும்போது: Data pages-ல் EmpID order-ல் rows store ஆகும் (1,2,3,...50). இது HEAP அல்ல — physically sorted linked list. Page 1:101 → 1:102 → 1:103 → 1:104 → 1:105 doubly linked.
03
Data Pages — dbo.Departments (20 rows → 2 Pages)
2nd table-க்கான data pages — same skeleton, வேற page numbers
Departments table: 20 rows ÷ 10 rows per page = 2 pages
Page 1:201 → DeptID 1–10  |  Page 1:202 → DeptID 11–20
இந்த table-ல் DeptID PK ஆக இருக்கு → Clustered Index on DeptID → sorted order.
⚠️ இரண்டு வேற tables, இரண்டு வேற page ranges:
Employees: pages 1:101 → 1:105 (FileID 1, PageNum 101 to 105)
Departments: pages 1:201 → 1:202 (FileID 1, PageNum 201 to 202)
SQL Server allocates pages from the same data file but tracks them separately via IAM (Index Allocation Map).
04
Clustered Index Pages — Employees
CREATE UNIQUE CLUSTERED INDEX on EmpID → எத்தனை index pages வரும்? என்ன skeleton?
CREATE UNIQUE CLUSTERED INDEX CIX_Employees_EmpID
ON dbo.Employees (EmpID ASC);
-- PRIMARY KEY automatically creates this!
Clustered Index Leaf Level = Data Pages themselves!
Leaf node-ல் separate pages இல்லை — data pages 1:101→1:105 இதுவே leaf nodes.
இதுக்கு மேல Non-Leaf (intermediate + root) pages மட்டும் create ஆகும்.
🏗️ INDEX PAGE ANATOMY — leaf-level இல்லாத upper level pages
Index Page 1:301 (Intermediate)
① PAGE HEADER
Page ID: 1:301
PageType: INDEX (2)
IndexID: 1 (clustered)
Level: 1 (intermediate)
② INDEX ROWS (Key + Page Pointer)
EmpID=1 → 1:101
EmpID=11 → 1:102
EmpID=21 → 1:103
EmpID=31 → 1:104
EmpID=41 → 1:105
③ FREE SPACE
Clustered Index Row Structure:
[Index Key (EmpID)] + [Page Pointer to child page]
No RID needed! Leaf level IS the data page. So intermediate nodes point directly to child pages.
50 rows → எத்தனை index pages?
Leaf (Data Pages): 5 pages (1:101–1:105) — already exist
Intermediate: 1 page (1:301) — 5 pointers, fits in one page
Root: 1 page (1:400) — points to intermediate
Total extra pages: 2 (intermediate + root)
Rule: When is intermediate needed?
Root மட்டும் போதுமா? 5 pages fit in root → technically 2 levels.
Real SQL Server-ல் data பெரிதாக இருந்தால் 3+ levels வரும்.
📇 CLUSTERED INDEX PAGES — Employees table
💡 Clustered Index = Data ஒரே structure! Leaf nodes = data pages. Index pages-ல் rows scan செய்ய வேண்டாம் — directly data pages-க்கு pointer மட்டும் வைக்கும்.
05
B-Tree Seek — Clustered Index on Employees
Root → Intermediate → Leaf(=Data Page) → Row Found! — interactive seek
🌳 B-Tree எப்படி உருவாகுது?
1. முதலில் data pages (leaves) உருவாகும்
2. Page count > threshold → intermediate page create
3. Intermediate count > threshold → root create
4. Root எப்போதும் ஒரே ஒரு page!
📐 எந்த key இந்த page-ல் போகும்?
Intermediate row = ஒவ்வொரு child leaf page-ல் முதல் key
Page 1:101 first row = EmpID 1 → "1 → 1:101"
Page 1:102 first row = EmpID 11 → "11 → 1:102"
...and so on
🔍 Seek எப்படி நடக்கும்?
EmpID=35 தேடுகிறோம்:
Root: 35 ≥ 31 → go to 1:301
Intermediate: 31≤35<41 → go to 1:104
Data Page 1:104: scan → EmpID=35 found!

06
Non-Clustered Index — Same Table, Different Index
CREATE INDEX on Salary column → separate index structure + RID lookup to data pages
CREATE NONCLUSTERED INDEX IX_Employees_Salary
ON dbo.Employees (Salary ASC);
-- Data pages unchanged! Separate index pages created.
🟢 Clustered Index (EmpID)
Leaf level = Data pages (actual rows)
Seek → Root → Inter → Data Page (leaf)
No extra lookup needed
3 I/Os to find any row
🔵 Non-Clustered Index (Salary)
Leaf level = Salary + RID pointer
Seek → Root → Inter → Leaf → RID → Data Page
Extra lookup to heap/clustered
4 I/Os to find any row

② NCI INTERMEDIATE
③ NCI LEAF → RID
④ DATA PAGE FETCH
✅ FOUND
