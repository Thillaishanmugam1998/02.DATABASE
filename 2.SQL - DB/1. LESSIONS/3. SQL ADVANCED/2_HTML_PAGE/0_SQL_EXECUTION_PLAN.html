02
Index Scan vs Index Seek
Two different execution paths with different performance

Query Without Index (Table Scan)

SELECT * FROM Employees WHERE EmployeeID = 5;
⚠️ Execution Plan: TABLE SCAN
Input: Employees Table
All 1000 rows loaded from disk
Reads: 8 pages | Rows: 1000
↓
Filter: EmployeeID = 5
Check each row one by one
Comparisons: 1000 | Found at: row 5
↓
Output
Return 1 matching row
Result Rows: 1 | Time: 5ms

Query With Index (Index Seek)

CREATE CLUSTERED INDEX PK_Employees ON Employees(EmployeeID);
SELECT * FROM Employees WHERE EmployeeID = 5;
✓ Execution Plan: INDEX SEEK
Seek in Root Node
Navigate B-Tree root (EmployeeID 1-500)
Read: 1 page | Decision: Go right
↓
Seek in Branch Node
Navigate intermediate node (EmployeeID 1-250)
Read: 1 page | Decision: Go left
↓
Seek in Leaf Node
Binary search for EmployeeID = 5
Read: 1 page | Found at: leaf position 5
↓
Output
Return 1 matching row
Result Rows: 1 | Time: 0.5ms (10x faster!)
❌ Table Scan (Without Index)
Pages Read: 8 (all of them)
Comparisons: 1000
Time: 5ms
CPU: High
I/O: Maximum
✓ Index Seek (With Index)
Pages Read: 3 (Root → Branch → Leaf)
Comparisons: ~10 (binary search)
Time: 0.5ms
CPU: Minimal
I/O: Minimal
💡 Performance Impact
Table Scan: O(n) - scans every row until found
Index Seek: O(log n) - uses B-Tree navigation to jump directly to row

For 1,000,000 rows: Table Scan = 1M comparisons vs Index Seek = ~20 comparisons!
03
JOIN Execution Plans
Nested Loop vs Hash Join – Different strategies for combining tables

Join Query

SELECT e.Name, d.DepartmentName
FROM Employees e
JOIN Departments d ON e.DepartmentID = d.DepartmentID
WHERE d.Location = 'NYC';

Nested Loop Join (Small Dataset)

⚡ Execution: Nested Loop Join
Outer Input: Departments (NYC)
Filter departments where Location = 'NYC'
Rows: 3 (NYC departments)
↓ For each department...
Inner Input: Employees (per iteration)
For each NYC department, seek employees with matching DeptID
Iterations: 3 | Seeks: 3 | Total Lookups: 3
↓
Join Operation
Combine rows: Each NYC dept + its employees
Output Rows: 287 (3 depts × ~96 employees each)

Hash Join (Large Dataset)

⚙️ Execution: Hash Join
Build Phase: Create Hash Table
Load Employees into memory, hash on DepartmentID
Hash Table Size: ~50MB | Memory Used: High
↓
Probe Phase: Match Departments
For each Department, lookup in hash table (O(1) lookup)
Lookups: 3 | Hash Collisions: 0 | Time: Instant
↓
Output Joined Rows
All matches from hash table
Output Rows: 287 | Total Time: 10ms
Nested Loop Join
Best For: Small outer input
Memory: Minimal
I/O: Multiple seeks
Complexity: O(n × m)
Pros: Efficient for small sets
Cons: Slow for large outer input
Hash Join
Best For: Large datasets
Memory: High (both tables loaded)
I/O: Two full scans
Complexity: O(n + m)
Pros: Fast for huge sets
Cons: Uses more memory
💡 When SQL Server Chooses
The Query Optimizer analyzes table sizes, indexes, and statistics to choose the best join strategy. For 10 employees: Nested Loop. For 10,000 employees: Hash Join. The execution plan adapts to your data!
04
Query Compilation & Caching
First run compiles the plan, subsequent runs reuse it

Query Plan Compilation Flow

❌ First Execution (No Cache)
1. Parse: Check syntax (1ms)
2. Bind: Resolve names (2ms)
3. Optimize: Find best plan (8ms)
4. Compile: Generate executable (3ms)
5. Execute: Run plan (10ms)
Total: 24ms
✓ Subsequent Runs (Cached)
1. ✓ Parse: (cached)
2. ✓ Bind: (cached)
3. ✓ Optimize: (cached)
4. ✓ Compile: (cached)
5. Execute: Run plan (10ms)
Total: 10ms (58% faster!)
💡 Plan Cache Strategy
SQL Server caches execution plans in memory. The first time you run a query, it spends time optimizing. Subsequent identical queries skip compilation and jump straight to execution. This is why reusing queries and parameterized queries are important!

Pro Tip: Use sp_executesql or parameterized queries to maximize plan reuse and avoid plan cache bloat.
